## 1 · 执行摘要

**一句话**:这是一个**工程内核极其成熟、但产品层基本缺失**的单人/LLM-in-loop/本地挂载/绑定 Claude Code 的 deck 工厂。

**最值得肯定的核心判断**:把 LLM 的输出空间从「自由 HTML」收敛到「受约束的 JSON schema」,再确定性渲染 + 硬闸门校验(`deck.json → render-deck.py → validate.py 三重闸门`)——这个"schema 驱动的编译器"思路是对的,也正是"组织级品牌一致性"需要的地基。**不要破坏这个内核。**

**最大的两类债务**:
1. **单一真相源(SSOT)被侵蚀**:4 档字号 `{16,24,28,48}`、variant→字段契约、规则码/阈值/opt-out 词表,都在 3–5 个文件(CSS / 校验器 Python / 校验器 JS / deck-cli / SKILL.md prose)里**手抄**,且已经实际漂移(deck-cli 缺 2 个已发布 variant;HERO 集合 Python 与 JS 成员不一致)。
2. **产品层缺失**:要"全员干掉 PPT",`无 HTML→pptx 导出`、`非工程师零创作路径(硬挂载闸门)`、`协作/评审/托管/遥测全无`、`无实时数据绑定` —— 这些才是采用率的真正天花板,而不是引擎能力。

**成本面**:`SKILL.md = 420KB ≈ 105K tokens`,**每次生成 deck 都全量进上下文**,是规模化后最大的边际成本;`pptx-to-html/` 在磁盘上占 **797MB**(689MB 未压缩样例图 + 108MB venv),且仓库**没有 `.gitignore`** 兜底。

## 3 · 内核优势(改造时**不要破坏**这些)

- **schema 即契约,真落地**:`render-deck.py main()` 先 `validate-deck.py --strict`(闸门1)→ 渲染 → `validate.py`(闸门3)。三重闸门是代码强制,不是文档口号。
- **渲染器两段式干净**:微型模板引擎(`render_template` 仅 2 个正则 `{{安全}}`/`{{{原始}}}`)+ per-layout `ENRICHERS` 派发表;命令式复杂度全隔离在具名 enricher 里。
- **文本转义单一路径** `_esc_br`(`\n→<br>` + HTML 转义),`{{ }}` 默认转义、`{{{ }}}` 罕见可信例外,有注释说明意图。
- **校验器又全又快**:35 个静态 audit + 16 个 Playwright 视觉 audit;**单次 Chromium 启动 + 单次 page.evaluate** 跑完所有视觉断言(28 页 ~2s);Playwright 缺失时优雅降级。
- **运行时 JS 是范本**:单 `AbortController` 收口所有监听器、单文档级 `ResizeObserver`(rAF 合并)、单 `MutationObserver`、`content-visibility:auto`(实测省 ~600ms 首屏)、`prefers-reduced-motion`、演讲器键位适配。
- **交付契约成熟**:禁止"单 linked HTML"裸文件 + 3 种交付形态(inline / editable-zip / hosted-URL)+ 一键启动器,真正解决了"接收方看到乱码"。
- **治理基元已具雏形**:`business-rules.yaml` 把技术规则翻译成业务语言供非工程师维护;`PPT-LIBRARY-MAPPING` 显示历史 122 页母版库 ~90% 已能落到现有 schema layout;CI 已存在并诚实(default + --strict + 单测 + 非-Cowork 环境自测 preflight)。

## 5 · 迭代路线图(战略 initiative)

> 工程内核(schema/validator)留作治理核心;在其外包一层**托管、自助、协作、pptx 桥接**的产品层。按阶段推进,每阶段可独立交付价值。

### 阶段 0 · 地基硬化(1–2 周,清 P0 卫生 + 成本)
- **R-00** 执行 F-06(.gitignore + 样例瘦身 + venv→requirements)、F-30/F-31(SKILL.md 拆 core + 移历史到 CHANGES.md)、F-01/F-12/F-16/F-14/F-17(快速制胜批)。**先做 F-09 测试骨架**,为后续模块化兜底。

### 阶段 1 · 解锁采用(关 2 个 critical 阻塞)
- **R-01 · HTML→PPTX 桥**(关 F-36):`deck.json → python-pptx`。Phase 1 每页 PNG 拍平的 .pptx 即时满足"我要文件";Phase 2 形状级可编辑导出。**最高杠杆。**
- **R-02 · 托管自助创作面**(关 F-37 + F-38):服务端跑同一 `deck.json→render→validate` 引擎,飞书登录,模板画廊 + 浏览器编辑,无挂载无终端。复活退役的服务端编辑器为托管产品。编辑器作用于 deck.json(非 raw DOM)。**触及非工程师多数的总闸门。**

### 阶段 2 · 留存与规模(让 deck 胜过 PPT 的工作流)
- **R-03 · deck.json 感知的结构化编辑器**(关 F-38):增删页/换 layout/传图,全部 patch 结构化模型 + 重校验,使自助编辑**强化**而非绕过治理核心。
- **R-04 · 协作 + 评审层**(关 F-39):slide-key 锚定评论 + draft/in-review/approved 状态机 + 飞书通知。
- **R-05 · 第一方托管分发**(关 F-40):一条命令发布到带权限的稳定 URL,移动友好,就地更新=唯一规范链接;解锁浏览遥测。
- **R-06 · 遥测与采用仪表盘**(关 F-42):渲染时 `{layout,variant,slide_count}` 计数 + opt-in 浏览 beacon → 飞书 Base 仪表盘。指导维护 + 向领导证明进度(保住项目预算)。

### 阶段 3 · 决定性差异化 + 治理
- **R-07 · 实时数据绑定**(关 F-43):deck.json 字段绑定飞书 Base/Sheets,render 前刷新 → 周报一键更新。**最高频报告品类的决定性 PPT 杀手。**
- **R-08 · 批量 PPT 入库**:把 pptx-to-html 子技能 + PPT-LIBRARY-MAPPING(~90% 已覆盖)产品化为批量导入器,一次跑迁一整个文件夹,~10% 需手工的出 triage 报告。
- **R-09 · 字体与品牌治理硬化**(关 F-44/F-46/F-47):自托管 subset webfont(或全链路统一 Noto)使像素一致;授权资产移单独权限库 + 外部使用检查 + `BRAND-LICENSE.md`;治理文件随 skill 分发。
- **R-10 · 单一真相源工程债**(关 F-02/F-13/F-21/F-03/F-10/F-11):tokens 单一源 + 跨语言共享配置 + 规则元数据生成文档 + validate.py 拆包 + CSS 真解析器。**在加下一批 layout/规则之前做**,否则技术债随报告类型增多复利。

### 建议执行顺序
`阶段0(地基) → R-01(pptx 桥,即时解阻塞) → R-02(托管+编辑,触及多数人) → R-05+R-06(托管成熟后顺带) → R-07(高频报告决定性优势) → R-03/R-04(协作深化) → R-10(债务,持续穿插) → R-08/R-09(规模化收尾)`
