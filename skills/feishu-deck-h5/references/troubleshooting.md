# troubleshooting — feishu-deck-h5 reference
> 从 SKILL.md 拆出(F-30 瘦身)· 何时读:渲染坏了但 validator 没指出时:症状→修

## Failure modes & fixes

| Symptom                                | Likely cause                                         | Fix |
|----------------------------------------|------------------------------------------------------|---|
| Slide displays at top-left, tiny       | Forgot to wrap `.slide` in `.slide-frame`            | Add the wrapper. |
| Indicator + toggle don't appear        | Missing `<script src="assets/feishu-deck.js">`       | Add it (or inline). |
| Mobile shows huge whitespace           | Viewport meta tag missing                            | Add `<meta name="viewport" ...>`. |
| Title overflows past edge              | Content too long for 1920 px canvas                  | Cut content. Don't shrink type below 24 px. |
| Card heights misaligned                | Card content imbalanced                              | Add a 1-line `<br>` to short titles. Cards are min-height:400. |
| Stats column rule on first column      | Default CSS leaks                                    | First column has `border-left:0` already — check overrides. |
| Two accents on one slide               | Forgot to set `data-accent` on slide level           | Set `data-accent="teal"` on the `.slide` element only. |
| Quote glow too strong                  | Custom background overrides `--fs-grad-glow-blue`    | Don't override `.slide[data-layout="quote"]` background. |
| `lift-slides.py` dies `FileNotFoundError` writing DEST (doubled `skills/feishu-deck-h5/runs/...`) | RELATIVE dst resolved against the symlinked skill root, not where `new-run.sh` made the run | Pass the ABSOLUTE run path `new-run.sh` printed for src, DEST `deck.json`, and OUTPUT_DIR. (Tool now fails fast with this hint before parsing the source.) |
| Lifted slide: many `R-VIS-DEAD-RULE` errors, layout collapsed to vertical | `--shake` recovered the source's per-key head CSS, which included shared-stylesheet cruft targeting elements absent on this slide | Confirm it's source cruft (grep source for key+selector), then prune only rules whose leaf selector matches no body element; keep the framework layout block. |
| Render says `PASS / errors 0` yet a card's text visibly spills past its box / over the bottom strap (esp. `raw` + `lifted` slides) | Default gate is `--no-visual` (static only); Playwright geometry audits ran as a NON-BLOCKING advisory (F-253). The card box had grown via `min-height` / legacy `align-self:stretch` so `scrollHeight≈clientHeight` (no *card*-level overflow) while the box itself overflowed its `.grid` and the **centered** content overlapped the sibling strap. `R-OVERLAP` only compares same-container direct children, so a cross-container spill is invisible to it. | **Fixed 2026-06-06**: `render-deck.py` now PROMOTES the HARD geometry errors (`R-VIS-CARD-OVERFLOW` / `R-OVERLAP` / `R-OVERFLOW` / `R-VIS-BAND-COLLIDE`) out of the advisory into a BLOCKING gate on full `/runs/` renders (override `DECK_ALLOW_GEOM_OVERFLOW=1`). When you scope-render / `--quick` / override, **measure the right thing**: container `el.scrollHeight > clientHeight` AND last content child `.bottom` vs the next sibling's `.top` — NOT just the card's bounding box (which `min-height` can inflate to hide the spill). |
| EVERY page's ad-hoc Playwright shot hangs ever since one page embedded a live external iframe (larkoffice doc / dashboard): `goto(wait_until='load')` hits the 30s timeout, and even after `domcontentloaded` the `screenshot()` stalls on "waiting for fonts to load…" | All frames share one DOM, and a live embed never settles (long-polling, streaming webfonts) — so the whole deck stops being screenshot-able the naive way. Settled-state tooling vs live content is an inherent conflict, not a flaky env. | Use `render-deck.py --scope N`'s deck-log auto-snapshot (handles it), or `assets/shoot-page.py <index.html> <N>` for ad-hoc shots — it route-ABORTS http(s) by default so the live panel renders empty and the shot lands in ~2s, deterministic. `--allow-external` only when you're shooting the live embed itself. Never hand-roll `wait_until='load'` against such decks. See prototype-embed.md F-304 for whether a live iframe belongs in the deck at all. |
| Same deck PASSES on one machine but FAILS (overflow / balance / title-position) on another — esp. a cloud Linux box vs your Mac | Visual-audit geometry is measured against the CJK **font metrics of the host that ran it**. The framework CJK face `方正兰亭黑 Pro GB18030` is a **locally-licensed** font with **no @font-face / no bundling** (F-283 step 1), so a host without it falls back (Noto / PingFang / tofu) and every glyph width — hence every overflow / balance / title number — shifts. A green verdict on one machine is not portable to another with a different CJK face. | Run `bash assets/preflight.sh` (or `check-mira.sh`) on **both** hosts and compare the `CAPABILITY cjk-font:` line; and compare `effective_cjk_font` in `validate.py <html> --json` — it stamps each verdict with the CJK face that run actually painted. If they differ, the geometry diff is the font, not the deck: install the master font (or the same fallback) on both, or treat the verdicts as non-comparable. (Full @font-face packaging so geometry is host-independent = F-283 **B version**, awaiting a go decision — not yet shipped.) |
| User keeps saying "黑边 / 背景黑 / 还是不对" on a raw/**iframe** slide, but your `shoot-page.py` shot looks fine (no black). **#27 cost ~6 rounds on exactly this.** | `shoot-page.py` renders a fixed **16:9** viewport (1920×1080), so it NEVER shows letterbox. The "黑边" is the **letterbox** top/bottom (or side) bars — painted by `.slide-frame:has(> .slide[data-slide-key="K"])`'s background — which **raw/iframe slides don't auto-inherit**, so it defaults to black. Your render ≠ the user's taller-than-16:9 window; re-confirming with the same 16:9 tool is the trap. | (1) Reproduce their window: `shoot-page.py <html> N --viewport 1600x1000` (any **non-16:9**) to actually SEE the bars. (2) On a raw/iframe slide, diagnose 黑边 as **letterbox first**, not the slide bg — fill it: `.slide-frame:has(> .slide[data-slide-key="K"]){background:<deck bg>!important}` (+ a `::after` glow to match `mix-glow` pages). (3) When the user says "还是不对" but your shot looks fixed → your shot ≠ their view: change `--viewport`, or ask for **their** screenshot — never keep re-confirming. (4) The deck's real content-bg is near-**black** at center/bottom (vibrancy is the content, not the bg) — sample `#26` before assuming "vibrant blue". |
| Page-turn (按翻页) stuck / deck won't advance while on an **embedded-iframe** slide (esp. after you added a scrollable region) | The iframe captures the deck's nav input: a scrollable child (`overflow-y:auto`) eats arrow keys / wheel; clicking the iframe steals focus from the deck's keyhandler (which lives on the parent). | Set the **iframe element `pointer-events:none`** — static mockups need no interaction, so clicks/page-turn pass straight through to the deck — and keep iframe content `overflow:hidden`. Verify in Playwright: `goto #N → click the dashboard → press ArrowRight → the `.slide-frame.is-current` slide-key MUST change`. To reveal hidden iframe content (a report below the fold) do it with a one-shot CSS transform / `justify-content`, **never** a live `overflow:auto` scroll region. |
